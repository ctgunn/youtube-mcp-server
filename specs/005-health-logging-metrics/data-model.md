# Data Model: FND-005 Health, Logging, Error Model, Metrics

## Entity: RequestContext
Description: Per-request operational context propagated across handling, logging, error mapping, and metric recording.

Fields:
- `requestId` (string, required): Correlation identifier from client payload or generated by server.
- `path` (string, required): Requested endpoint path.
- `methodName` (string, optional): MCP method name when path is `/mcp`.
- `toolName` (string, optional): Tool identifier when MCP call targets `tools/call`.
- `startedAt` (timestamp, required): Request start time for latency calculation.

Validation rules:
- `requestId` MUST always be present before response emission.
- `toolName` MUST be populated only for tool call requests with a valid tool target.

## Entity: HealthStatus
Description: Liveness/readiness output payload consumed by operators and probes.

Fields:
- `status` (string, required): `ok` for liveness; `ready` or `not_ready` for readiness.
- `checks` (object, optional): Named readiness checks with `pass`/`fail` values.
- `reason` (object, optional): Non-sensitive failure reason when readiness is not ready.

Validation rules:
- Liveness responses MUST provide `status` with healthy value.
- Not-ready responses MUST include non-sensitive machine-readable reason metadata.

## Entity: ErrorEnvelope
Description: Normalized client-visible error object.

Fields:
- `code` (string, required): Stable machine-readable error class.
- `message` (string, required): Sanitized human-readable summary.
- `details` (object|null, optional): Additional structured context without internal trace leakage.

Validation rules:
- Every error response MUST include `code`, `message`, and `details` keys.
- Internal stack traces or secret values MUST NOT appear in `message` or `details`.

## Entity: LogEvent
Description: Structured operational log record produced per handled request.

Fields:
- `timestamp` (string, required): Event time in standard serialized format.
- `severity` (string, required): Log severity level.
- `requestId` (string, required): Correlation ID for request lifecycle tracing.
- `path` (string, required): Endpoint path.
- `toolName` (string, optional): Tool target when relevant.
- `status` (string, required): Success/error outcome classification.
- `latencyMs` (number, required): End-to-end request latency in milliseconds.

Validation rules:
- All handled requests MUST emit at least one log event with required fields.
- `latencyMs` MUST be non-negative.

## Entity: MetricSample
Description: Aggregated metric observation for request volume and latency.

Fields:
- `endpoint` (string, required): Endpoint dimension (`/healthz`, `/readyz`, `/mcp`, or not-found path class).
- `toolName` (string, optional): Tool dimension for `tools/call` requests.
- `outcome` (string, required): `success` or `error`.
- `count` (integer, required): Incremental request count.
- `latencyMs` (number, required): Observed request latency input to percentile summaries.

Validation rules:
- Every handled request MUST increment request count metrics.
- Outcome counts MUST be segmentable into success and error paths.
- Latency observations MUST support percentile derivation (p50/p95).

## Relationships

- `RequestContext` is created per request and links to one `LogEvent` and one or more `MetricSample` observations.
- `ErrorEnvelope` is produced from request handling failures and carries `requestId` correlation via response metadata.
- `HealthStatus` is produced by health/readiness paths and contributes corresponding `MetricSample` and `LogEvent` entries.

## State Transitions

1. Request received -> build `RequestContext` (assign/generate `requestId`).
2. Route handling by path/method/tool -> compute success/error outcome.
3. Generate response payload (`HealthStatus` or MCP envelope with optional `ErrorEnvelope`).
4. Emit `LogEvent` with final outcome and latency.
5. Record `MetricSample` counts and latency observation.
