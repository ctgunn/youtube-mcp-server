# Data Model: Automated Hosted Network Bootstrap Reconciliation

## HostedDeploymentRun

- **Purpose**: Represents one end-to-end execution of the reviewed hosted deployment chain for a specific revision and environment.
- **Fields**:
  - `revision_ref`: source revision being deployed
  - `automation_surface`: `cloud_build_primary` or `github_actions_fallback`
  - `target_environment`: hosted environment identifier such as `staging` or `prod`
  - `stage_order`: ordered list of expected workflow stages
  - `overall_result`: `pass`, `fail`, or `incomplete`
  - `first_failed_stage`: first workflow stage that did not pass
  - `artifact_set`: references to exported Terraform outputs, deployment record, and hosted verification evidence
- **Validation Rules**:
  - `stage_order` must preserve `quality_gate -> image_publish -> infrastructure_reconcile -> terraform_output_export -> deploy -> hosted_verification`.
  - A run cannot report `pass` unless every required stage passes.
  - A run that fails before deploy must still preserve enough artifact evidence for operator diagnosis.
- **Relationships**:
  - Owns one `NetworkBootstrapStage`.
  - Owns one `BootstrapInputSet`.
  - Produces one or more `StageOutcomeRecord` entries.
  - Is interpreted through one `BootstrapFailureClassification`.

## NetworkBootstrapStage

- **Purpose**: Captures the portion of infrastructure reconciliation that proves the managed hosted network layer is part of the rollout path before application deployment begins.
- **Fields**:
  - `stage_name`: `infrastructure_reconcile`
  - `managed_network_required`: whether the targeted environment depends on the managed hosted network layer
  - `reconcile_scope`: infrastructure responsibilities expected during the stage
  - `network_references_exported`: whether the managed network outputs are available after reconciliation
  - `result`: `pass`, `fail`, or `not_required`
  - `failure_reason`: operator-visible explanation when the stage fails
- **Validation Rules**:
  - If `managed_network_required=true`, the stage must not be treated as passed unless managed network reconciliation succeeds.
  - A failed bootstrap stage must block deploy and hosted verification from being treated as successful rollout stages.
  - The stage must remain part of the existing infrastructure reconciliation contract rather than becoming an undocumented side effect.
- **Relationships**:
  - Belongs to one `HostedDeploymentRun`.
  - Consumes one `BootstrapInputSet`.
  - Produces one `StageOutcomeRecord`.

## BootstrapInputSet

- **Purpose**: Represents the one-time external inputs that must exist before the recurring hosted deployment pipeline can run successfully.
- **Fields**:
  - `automation_inputs`: repository and cloud access prerequisites required by automation
  - `terraform_environment_inputs`: environment-specific infrastructure inputs supplied through the reviewed Terraform variable file
  - `operator_managed_secret_values`: required secret values that must be populated outside automation
  - `network_provisioning_owned_by_automation`: whether recurring hosted network provisioning remains inside the pipeline
  - `status`: `complete`, `missing`, or `ambiguous`
- **Validation Rules**:
  - `network_provisioning_owned_by_automation=true` is required for the supported hosted path covered by this feature.
  - Missing automation inputs or operator-managed secret values must fail the run before rollout is reported as successful.
  - Terraform environment inputs may describe network names or ranges, but they must not imply a separate recurring manual network provisioning workflow.
- **Relationships**:
  - Referenced by one `HostedAutomationBoundary`.
  - Evaluated before or during one `HostedDeploymentRun`.

## StageOutcomeRecord

- **Purpose**: Describes the operator-visible result of one deployment stage and the artifacts or remediation associated with it.
- **Fields**:
  - `stage_name`: workflow stage identifier
  - `result`: `pass`, `fail`, or `incomplete`
  - `summary`: concise operator-facing outcome
  - `artifact_path`: optional artifact generated by the stage
  - `remediation_boundary`: whether remediation belongs to bootstrap inputs, network reconcile, deploy, or verification
- **Validation Rules**:
  - Failed stages must produce enough summary detail for an operator to classify the failure boundary.
  - `artifact_path` should be present when the stage emits reviewable evidence.
  - Stage names must align with the ordered hosted deployment chain.
- **Relationships**:
  - Belongs to one `HostedDeploymentRun`.
  - Supports one `BootstrapFailureClassification`.

## BootstrapFailureClassification

- **Purpose**: Groups operator-facing failure outcomes into the minimum categories needed to decide where remediation belongs.
- **Fields**:
  - `classification`: `bootstrap_input_failure`, `network_reconcile_failure`, `deployment_failure`, or `hosted_verification_failure`
  - `trigger_stage`: stage where the failure becomes visible
  - `operator_owner`: `automation`, `operator`, or `shared`
  - `diagnostic_artifact`: primary artifact or output used for diagnosis
  - `release_blocking`: whether the failure prevents rollout from being reported as successful
- **Validation Rules**:
  - Every failed run must map to exactly one primary classification.
  - `network_reconcile_failure` must remain distinct from `deployment_failure`.
  - All classifications for this feature are release-blocking.
- **Relationships**:
  - Interprets one or more `StageOutcomeRecord` entries.
  - Applies to one `HostedDeploymentRun`.

## HostedAutomationBoundary

- **Purpose**: Captures the documented line between what the checked-in hosted deployment chain reconciles automatically and what remains outside the recurring run.
- **Fields**:
  - `primary_pipeline`: automatic hosted deployment surface
  - `fallback_pipeline`: manual fallback surface
  - `automation_managed_scope`: recurring responsibilities owned by the pipeline
  - `externally_managed_scope`: one-time prerequisites outside the recurring pipeline
  - `local_path_excluded`: whether local execution is kept outside hosted bootstrap requirements
  - `documentation_state`: `aligned`, `stale`, or `missing`
- **Validation Rules**:
  - The automation-managed scope must include managed network reconciliation for the supported hosted path.
  - The externally managed scope must exclude recurring manual network provisioning.
  - `local_path_excluded=true` is required to preserve the local-first development contract.
- **Relationships**:
  - References one `BootstrapInputSet`.
  - Applies across one or more `HostedDeploymentRun` records.

## State Transitions

### Hosted Deployment Run

1. `planned` -> automation surface, stage order, and target environment are identified.
2. `bootstrap-validated` -> required one-time bootstrap inputs are present.
3. `network-reconciled` -> infrastructure reconciliation completes for the managed hosted network layer.
4. `outputs-exported` -> Terraform outputs are exported for downstream deployment and verification.
5. `deployed` -> application rollout completes and emits a deployment record.
6. `verified` or `failed` -> hosted verification succeeds or the run terminates with a classified failure.

### Bootstrap Failure Classification

1. `unclassified` -> failure is detected but not yet assigned to an operator-visible boundary.
2. `bootstrap_input_failure` -> one-time external prerequisites were missing or unusable before reconcile.
3. `network_reconcile_failure` -> managed network reconciliation failed during infrastructure reconciliation.
4. `deployment_failure` -> application rollout failed after successful bootstrap and output export.
5. `hosted_verification_failure` -> post-deploy hosted verification failed after rollout.
